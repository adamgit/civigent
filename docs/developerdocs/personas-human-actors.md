These are the user-personas we use when considering design/architecture changes: what do these users need, what do they expect, how would they react to what they see? ... etc

**Raj, the Engineering Lead** — Owns the architecture docs and API specs. Treats the Knowledge Store as his team's source of truth. He'll open a doc, sketch a rough outline of a new system design, then tell his agent to "fill in the implementation details for sections 3 through 7" while he moves on to code review. He checks the Mirror panel obsessively because he wants to publish his skeleton before the agent starts proposing against stale structure.

**Sophie, the Product Marketing Manager** — Maintains messaging guides, competitive battlecards, and launch briefs. Zero terminal experience, but she's learned to talk to her agent in precise natural language. She writes the "what we want to say" sections herself and directs her agent to update all the supporting data — market size numbers, competitor feature tables, analyst quotes. She gets frustrated when her agent is blocked because she was editing the intro paragraph twenty minutes ago.

**Dave, the Senior Backend Engineer** — Lives in his editor, rarely opens the Knowledge Store UI directly. He points his agent at docs and says "update the runbook to reflect the new deployment pipeline." He trusts the agent completely for operational docs but would never let it touch the architecture decision records he personally authored. He doesn't know what the heatmap is.

**Kim, the Content Strategist** — Owns the customer-facing docs, help center articles, and onboarding guides. She's the most prolific human editor in the system — dozens of small edits across many documents daily. She uses her agent as a copy editor and consistency checker: "read all the onboarding docs and make sure we're calling it 'workspace' not 'project' everywhere." She's the person most likely to hit section-too-long warnings because she inherited docs from a wiki migration.

**Amir, the Junior Developer Advocate** — Writes tutorials and integration guides. He co-writes with his agent in real time — he'll draft a paragraph, then say "now write the code sample for this step and the explanation after it." He's the heaviest user of the proposal-accept flow, reviewing and accepting agent proposals every few minutes. He occasionally rejects and says "try again, simpler."
