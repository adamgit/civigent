
**"ArchBot" — ClaudeCode + OpenClaw orchestration layer.** An engineering team's shared agent running via OpenClaw's task routing and memory system on top of ClaudeCode. It reads broadly across architecture docs and runbooks, builds a mental model of system dependencies, then submits multi-section proposals that update cross-references when one doc changes. It provides detailed justifications citing the source document that triggered each change. It's the most sophisticated agent in the system — and the most likely to hit aggregate impact thresholds because it touches many sections at once.

**"ContentPilot" — ClaudeCode + popular open-source agent skills framework (e.g., "AgentKit").** Kim and Sophie's marketing org installed AgentKit's "docs-maintenance" skill pack onto a shared ClaudeCode instance. It comes with pre-built prompts for style consistency sweeps, terminology enforcement, freshness checks ("find sections with dates older than 90 days"), and competitive data updates. It's well-behaved but generic — it never provides per-section justifications because the skill pack doesn't know about that feature. It uses `?replace=true` on every proposal because the skill pack assumes stateless operation.

**"Amir's Copilot" — Vanilla ClaudeCode, human-directed in real time.** Amir runs ClaudeCode in his terminal and tells it what to do, step by step. It reads a section, Amir reads the proposal, accepts or rejects, then gives the next instruction. It has no orchestration layer, no memory between commands, no multi-document awareness. It's the simplest agent in the system but produces the highest-quality proposals because Amir is reviewing every single one. It's also the agent most likely to hit 409 conflicts because Amir forgets he has a draft open and tells it to submit a new one.
